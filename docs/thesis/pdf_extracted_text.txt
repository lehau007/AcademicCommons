2.1 Objectives 
●​ Develop an Academic Knowledge Infrastructure system to support students in 
contributing, standardizing, and utilizing community learning materials (PDFs, slides, 
photos). 
●​ Develop an Agent Pipeline to automatically process uploaded documents, including: 
content extraction (OCR / Document Parsing) and text content normalization. 
●​ Operate an Automatic Agent with Human-in-the-Loop (HITL) model with multi-level

Reviewer and Admin verify and make the final decision. 
●​ Develop AI Skills to support learning: 
○​ Virtual Tutor (AI Tutor) using RAG with a source citation mechanism 
○​ Automatic Mindmap generation from learning material content 
○​ Automatic Mock Test system generating review questions from materials 
●​ Establish an AI Evaluation Framework combining automatic metrics (RAGAS) and 
Author Audit to evaluate the quality of answers and mitigate hallucination.

standardized course information. This is a deliberate design decision to ensure data quality and 
simplify the processing flow within the scope of the thesis.

4. EXPERIMENT DESIGN AND SYSTEM EVALUATION 
 
4.1 Target Audience for Experiment 
Approximately 5-10 students at the School of Information and Communication Technology 
will participate in the pilot study. 
 
4.2 AI Evaluation Framework 
The system employs a two-tiered evaluation strategy: automatic metrics (RAGAS) and 
Author Audit. The two chosen frameworks are RAGAS (primary tier) and DeepEval 
(supplementary tier). 
●​ RAGAS: Standardized metrics for RAG systems, natively integrated with the 
LangChain pipeline, using LLM-as-judge for automatic scoring. 
●​ DeepEval: Supplements the ability to write unit tests for specific error types and 
integrates CI/CD, suitable for automating evaluation during development. 
Tier 1 - Automatic Metrics (RAGAS) 
Run on the entire Evaluation Dataset (50-100 Q&A pairs). Metrics set: 
●​ Faithfulness: Is every claim in the answer supported by the retrieved context? This 
is the primary metric for measuring hallucination. 
●​ Answer Relevancy: Is the answer appropriate and complete for the question? 
●​ Context Precision & Recall: Quality of the retrieval step-are the retrieved passages 
truly relevant and sufficient to answer the question? 
●​ Citation Accuracy (self-developed): Does the returned citation source match the 
actual context used to generate the answer? 
Tier 2 - Author Audit (Human Evaluation on a small subset) 
Purpose: Not to replace RAGAS, but to check calibration-confirming if RAGAS truly reflects 
user-perceived quality, and detecting subtle errors missed by RAGAS. 
Scale: 20-30 questions randomly selected with stratification from the Evaluation Dataset 
(including easy, medium, and content-difficult questions). 
Executor: The thesis author acts as the auditor. To mitigate confirmation bias, the audit is 
conducted blind (without viewing the RAGAS score first), and results are compared 
afterward. 
Annotation form: For each question, the auditor assesses 3 dimensions: 
●​ Factual Correctness: Is the answer content factually correct? (Correct / Incorrect / 
Partial) 
●​ Hallucination Flag: Is there fabricated information not present in the original 
document? (Yes / No / Unsure) 
●​ Citation Valid: Does the returned citation source actually contain the information used 
to answer? (Valid / Invalid / Irrelevant) 
Method for Comparing Results: Calculate the Pearson correlation between the RAGAS 
Faithfulness score and the Factual Correctness of the Author Audit. If the correlation is low 
(< 0.6), analyze the cause and note it as a limitation. If it is high (> 0.6), use this result to 
reinforce the credibility of RAGAS in the report.

5. DOCUMENT EVALUATION PIPELINE 
 
A core component ensuring the quality of the knowledge base. The entire pipeline runs 
asynchronously via Message Queue and integrates a self-correction mechanism. 
 
5.1 Overview of 4-Agent Architecture 
When a document is uploaded, the system creates a Document Evaluation Job and pushes 
it to Redis + BullMQ. A Worker processes 4 agents sequentially: 
 
Agent 1 - Document Summarizer 
●​ Receives OCR'd and normalized document content. Task: summarize document 
content according to a standardized structure including fields: main topic, list of 
concepts/skills mentioned, language, format, and estimated OCR quality. The output 
is a standardized JSON. The Agent has a retry mechanism if the output is not 
schema compliant (maximum 3 times). 
Agent 2 - Course Context Aggregator 
●​ Receives course information from document metadata (course name, course code). 
Task: aggregate two sources of context information: 
○​ Course description and official syllabus saved in the system (from 
PostgreSQL). 
○​ List and statistics of documents already accepted for that course (from the 
Academic Document Base), aiming to detect duplicate content. 
●​ Output: JSON containing course_description, existing_topics_coverage, 
and duplicate_flag. 
Cold Start Handling: When a course does not yet have a Course Seed Document, Agent 2 returns 
empty context with the flag cold_start: true. Agent 4 receiving this flag will automatically: (a) 
downgrade the confidence score to MEDIUM, (b) explicitly state the reason in the Evaluation 
Report, and (c) change the default Recommendation to NEEDS_REVIEW instead of APPROVE - 
forcing manual review. Since the system uses an Admin-controlled, pre-populated list of official 
courses, the risk of cold start due to user input error is eliminated. The Admin must ensure a 
minimum Course Seed Document (course name, short description, list of topics) is present for a 
course before its documents can be fully processed 
 
Agent 3 - Internet Search Agent

●​ Based on the course name and university, this agent performs internet searches to 
gather reference information about the course from authoritative sources (department 
website, public syllabus, OpenCourseWare). Goal: supplement objective external 
context, helping the evaluation agent not be limited by internal data. Integration via 
Tavily Search API or SerpAPI. The Agent has a 15-second timeout and a graceful 
fallback if no results are found.

Output of Agent 4: 
●​ Scores across 3 dimensions (0-10 each) according to the rubric above. 
●​ Weighted Score = 0.4 × Relevance + 0.3 × Completeness + 0.3 × Quality. 
●​ Duplicate Flag: detects duplication with documents already in the repository. 
●​ Recommendation: APPROVE (Weighted ≥ 7.0 and no cold start) / 
NEEDS_REVIEW (5.0-6.9 or cold_start flag) / REJECT (< 5.0 or Duplicate = true). 
●​ Full Evaluation Report with detailed reasons, saved to PostgreSQL and displayed on 
the HITL Dashboard. 
5.2 Self-Correction Loop 
Each agent checks the output schema after each run. If the output is invalid (incorrect JSON 
format, missing mandatory fields): the agent retries with an additional prompt describing the 
specific error (maximum 3 times). If it fails after 3 attempts: the job is marked 
EVALUATION_FAILED, transferred directly to the Admin for manual processing. All retry 
logs are recorded for debugging and prompt improvement. 
 
5.3 Message Queue Integration 
The pipeline operates entirely asynchronously via Redis + BullMQ. Job status is updated in 
realtime: QUEUED → PROCESSING → EVALUATED → APPROVED / REJECTED. 
Multiple workers run in parallel, ensuring no blocking when many documents are uploaded 
simultaneously.

6.2 HITL Processing Flow 
●​ After the Document Evaluation Pipeline is complete, the AI Evaluation Report is 
routed to the Subject Reviewer responsible for the corresponding course (based on 
domain mapping in the DB). 
●​ APPROVE + Reviewer agreement → document is automatically entered into the 
Academic Document Base. 
●​ NEEDS_REVIEW → Reviewer is required to manually review and make a decision. 
●​ REJECT → Reviewer can override if there is a justifiable reason, along with a 
mandatory note. 
●​ If there is no Reviewer for that course, or no response after 48 hours → automatically 
escalated to Admin. 
●​ All decisions (Approve / Reject / Override) are fully logged, serving to improve Agent 
4 prompts and the rubric later on. 
6.3 HITL Dashboard Design 
●​ Reviewer Dashboard: List of documents awaiting review by course; AI Evaluation 
Report with detailed score and reasons; Approve / Reject / Request More Info 
buttons; comment form. 
●​ Admin Dashboard: System overview; manage Reviewers and domain assignment; 
process escalated documents and EVALUATION_FAILED jobs; manage Course 
Seed Documents; activity statistics.

8. LIMITATIONS AND FUTURE DEVELOPMENT 
 
8.1 Current Limitations 
●​ Language: Optimized for Vietnamese and English. 
●​ Formats: Unencrypted PDF, images (JPG/PNG), PPTX. Low-quality scanned PDFs 
may result in inaccurate OCR. 
●​ Mathematical formulas: The ability to recognize and render LaTeX requires further 
verification. 
●​ Experiment Scale: A pilot study of 5-10 students is insufficient for statistical 
conclusions; results are qualitative and illustrate feasibility. 
●​ Author Audit single-annotator: Inter-annotator agreement cannot be calculated; it is 
framed as a 'manual calibration check' instead of full human evaluation. 
●​ Subject Reviewer in pilot: Handled by the thesis author's group, no independent 
domain-specific Reviewers yet. 
●​ Dependence on external services: Gemini API, Groq API, Supabase, Tavily/SerpAPI. 
●​ Course Classification: The system operates based on an official course list 
predefined by the Admin. This is a design decision intended to simplify the document 
processing workflow. Limitations: Since some university courses may have different 
codes and titles despite sharing similar curricula, classification based strictly on 
specific course codes may be restrictive and could limit the potential for 
interdisciplinary content discovery. 
8.2 Future Directions 
●​ Expand support for mathematical formulas (MathOCR / Mathpix API). 
●​ Increase experiment scale to 50+ students for statistical results. 
●​ Expand Author Audit to multi-annotator with volunteer students, adding Cohen's 
Kappa calculation. 
●​ Develop additional AI Skills: Automatic Summarization and Document Diff. 
●​ Integrate RLHF-lite from user feedback to improve answer quality over time. 
●​ Build a feedback loop from HITL override to the pipeline: save 'disagreement cases' 
(AI wrong, human correct) as data to improve Agent 4's rubric and prompt. 
●​ Expand the target audience to the entire Hanoi University of Science and 
Technology.
